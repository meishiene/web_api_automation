# 企业级自动化测试平台系统架构规划

## 1. 文档目的

本文档用于定义当前项目从 **MVP 级 API 自动化测试工具** 演进为 **企业级自动化测试平台** 的整体方向，作为后续架构设计、模块拆分、迭代排期与开发实施的总纲。

本文档重点回答以下问题：
- 当前系统处于什么阶段
- 企业级目标平台应该具备哪些能力
- API 测试与 Web 测试如何统一纳入平台能力体系
- 接下来每个阶段应该优先做什么
- 如何以最小风险从当前代码逐步演进到企业级平台

## 2. 当前系统现状

### 2.1 当前已具备能力
- 用户注册、登录、刷新令牌
- 项目管理（含成员管理）
- 组织管理与跨项目成员治理基础能力
- API 测试用例管理（含断言规则/提取规则）
- API 套件管理与有序编排
- 单条与套件批量执行（含批次追踪）
- 环境与变量管理（项目级/环境级 + 变量组复用 + 敏感变量脱敏与受控读取）
- Web 用例管理、单用例执行、执行详情与产物路径回溯最小闭环
- API/Web 统一执行结果查询与展示（Execution Center，支持类型/状态/时间范围筛选与分页）
- 执行结果记录、批次明细查询与基础展示
- 前后端联通主链路（登录/项目/用例/执行）

### 2.2 当前技术形态
- 后端：`FastAPI + SQLAlchemy + SQLite + httpx`
- 前端：`Vue 3 + Vite + Vue Router + axios`
- 架构模式：前后端分离、单体应用、本地数据库持久化
- 认证方式：JWT Bearer + Refresh Token（最小 RBAC + 权限矩阵 + 资源级鉴权基础已落地）

### 2.3 当前定位判断
当前系统属于：

- **阶段定位：阶段 2 收尾期 + 阶段 3 收尾期 + 阶段 4 验收完成 + 阶段 5 验收完成 + 阶段 6 启动中（S6-06 已完成，阶段 1 暂停）**
- **能力重心：阶段 2/3 收尾稳定性治理 + 阶段 6 企业集成推进（完成 CI/CD + 通知 + 缺陷联动 + OAuth2 最小闭环，进入治理增强准备）**
- **成熟度：MVP 向平台化过渡期，尚不具备企业级完整能力**

## 3. 平台愿景与建设目标

## 3.1 平台愿景
建设一个统一的企业级自动化测试平台，支持 **API 测试** 与 **Web 测试** 两大核心能力，并围绕测试资产管理、执行调度、报告分析、权限治理和企业集成形成完整闭环。

## 3.2 目标平台能力范围

平台中长期目标应覆盖以下能力：
- 统一用户、组织、权限体系
- 统一项目、环境、变量、密钥管理
- API 自动化测试设计、执行与回归
- Web 自动化测试设计、执行与回归
- 套件、批次、计划、调度、队列、Worker 分发
- 统一执行报告、趋势分析、稳定性分析
- 与 CI/CD、消息通知、SSO、缺陷系统集成

## 3.3 平台建设原则
- **先平台底座，后功能扩张**
- **先统一模型，后增加模块**
- **API 与 Web 共用治理体系**
- **先单体清晰分层，再考虑服务拆分**
- **所有执行必须可追踪、可审计、可分析**
- **优先保证可演进性，不追求短期功能堆砌**

## 6. API 与 Web 测试统一架构设计

## 6.1 统一原则
API 测试和 Web 测试不应建设为两个完全独立的平台，而应复用一套统一底座：
- 统一项目
- 统一环境
- 统一变量
- 统一权限
- 统一调度
- 统一结果中心
- 统一报告体系

## 6.2 API 测试能力清单
- 请求方法、路径、Header、Body、Query、Path 参数管理
- 状态码断言
- 响应体断言
- JSONPath / JMESPath / Schema 校验
- 变量提取与链路传递
- 套件执行、批量执行、回归计划执行

## 6.3 Web 测试能力清单
- 页面访问
- 元素定位与操作
- 等待与同步机制
- 文本、属性、状态断言
- 页面跳转与流程校验
- 截图、视频、Trace 产物输出
- 多浏览器执行
- UI 回归报告展示

## 6.4 Web 测试引擎建议
Web 测试执行引擎建议优先采用 `Playwright`，原因如下：
- 浏览器自动化能力成熟
- 原生支持截图、视频、Trace
- 支持 Chromium、Firefox、WebKit
- 对现代前端页面兼容性更好
- 更适合作为企业级 UI 自动化测试底座

平台架构层应保留执行引擎抽象，不把实现永久绑定到单一技术上，未来可兼容 Selenium 或其他执行器。

## 7. 目标系统架构规划

## 7.1 近期架构形态
短期建议仍维持 **单体应用 + 清晰分层 + 模块化领域边界**，避免过早拆分微服务。

### 近期建议结构
- 前端平台：统一工作台与业务模块页面
- 后端单体：按领域分模块组织
- 数据层：单主库
- 执行层：先单机执行，再引入独立 Worker

## 7.2 中期架构形态
当 API 与 Web 能力完成平台化后，逐步引入：
- 调度器
- 队列系统
- 独立 Worker 节点
- 对象存储
- Redis 缓存

## 7.3 长期架构形态
长期可演进为分层平台架构：

- `gateway / api layer`
- `auth service`
- `project service`
- `test asset service`
- `execution service`
- `schedule service`
- `report service`
- `notification service`

此阶段是否拆分为独立服务，应基于团队规模、并发规模和运维能力决定，不建议在项目早期强行微服务化。

## 8. 技术演进建议

## 8.1 数据库
建议由 `SQLite` 迁移到企业级数据库：
- 推荐：`PostgreSQL`
- 可选：`MySQL`

原因：
- 更适合并发访问
- 支持更完整的数据管理能力
- 更利于企业环境部署与备份恢复

## 8.2 缓存与任务中间件
建议引入：
- `Redis`：缓存、分布式锁、轻量任务状态保存
- 队列机制：初期可基于数据库或 Redis，后期视规模演进为消息队列

## 8.3 Web 测试执行能力
- 推荐执行引擎：`Playwright`
- 需要配套：浏览器执行环境、产物目录管理、结果上传机制

## 8.4 工程化能力
必须逐步补齐：
- 数据库迁移工具
- 统一异常处理
- 统一错误码规范
- 结构化日志
- 自动化测试体系
- 配置分环境管理

## 13. 风险与注意事项

### 13.1 架构风险
- 过早追求微服务会显著增加复杂度
- 在领域模型未稳定前，不宜大规模拆服务

### 13.2 产品风险
- 如果未先建设统一环境、变量、执行模型，后续 API 与 Web 能力会出现重复建设
- 如果只做执行，不做结果治理，平台价值会停留在工具层面

### 13.3 实施风险
- Web 测试执行环境对资源消耗更高，需要提前规划 Worker 和执行机管理
- 报告、截图、视频、Trace 会带来较大的存储压力，需要尽早设计产物存储策略

## 14. 结论

当前项目已经完成了企业级自动化测试平台的最小种子版本，但仍处于非常早期的 MVP 阶段。

后续开发应遵循以下主线：
- 优先完成阶段 2/3 收尾与阶段 1 遗留风险托管
- 在已完成阶段 4/5 验收基础上推进阶段 6 企业集成落地
- 采用“单提供商先落地 + 适配层抽象”策略，逐步扩展 CI/CD、通知、缺陷与身份集成

从当前阶段看，最佳策略不是推翻重写，而是基于现有闭环能力逐步演进，最终形成一个统一承载 **API 测试 + Web 测试 + 调度执行 + 报告治理 + 企业集成** 的企业级自动化测试平台。
