# 仓库级开发约束（AGENTS）

## 1. 进度文档维护

- 当代码或文档改动影响到功能实现、模块状态、架构分层、路线图或项目阶段时，必须在同一轮改动中同步更新 `docs/project/project-progress.md`。
- 发生上述改动后，必须同时检查 `docs/architecture/企业级自动化测试平台系统架构规划.md` 是否需要同步修订，确保长期规划与实际代码一致。
- 每次符合条件的改动，都要更新 `docs/project/project-progress.md` 中的“最近更新记录”。
- 仅有占位模型、占位页面或未打通链路的能力，不得标记为“已完成”。

## 2. 文档与代码一致性

- 以 `app/` 与 `frontend/src/` 下的实际代码为准，不以口头计划替代事实状态。
- 进度评估应保守、可证据化，不夸大完成度。
- 新增重要模块时，除代码改动外，必须同步更新 `docs/` 下对应文档。

## 3. 代码规范要求

- 必须遵循现有代码风格与目录结构，优先小步、可审查、可回滚的改动。
- 命名需语义清晰，禁止无意义缩写和一次性临时代码残留。
- 不得引入与当前任务无关的大范围重构或顺带修改。
- 新增逻辑需补充必要的错误处理与边界处理，避免静默失败。

## 4. 测试门禁（强制）

- 每次涉及“逻辑或功能代码”改动，必须执行测试验证。
- 原则要求：先补充或更新测试，再实现或调整功能逻辑。
- 测试通过后再修改仓库代码（按该仓库流程理解为：测试通过后才允许将改动作为有效结果保留并交付）。
- 至少执行与改动直接相关的最小测试集；若改动影响公共模块，还需补充更高层级回归测试。
- 若测试失败，不得将该改动标记为完成，需继续修复直至通过，或明确记录阻塞原因与风险。

## 5. 提交流程约束

- 完成开发后，输出中必须包含：改动范围、执行的测试、测试结果、已知风险。
- 未执行测试时，必须明确说明未执行原因及建议的补测命令。
